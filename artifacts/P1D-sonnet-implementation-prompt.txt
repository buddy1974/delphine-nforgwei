TASK 3 — FIX PARENT-ORIGIN HANDSHAKE (BIDIRECTIONAL — read carefully)

Context: Outbound messages FROM BrandWorkspace already target previewOriginRef
(derived from DELPHINE_PUBLIC_SITE_URL) and are correct. The bug is the INBOUND leg:
OsPreview posts back to new URL(VITE_OS_URL).origin, which silently drops if it does
not exactly equal the real parent origin. Fix that leg via a handshake:

1. BrandWorkspace, on iframe load (onIframeLoad), sends:
   { type: "PREVIEW_INIT", origin: window.location.origin, editMode: currentEditMode }
2. OsPreview listens for PREVIEW_INIT and stores the TRUSTED parent origin from
   event.origin (validate it equals the configured OS origin; keep VITE_OS_URL origin
   as fallback only).
3. OsPreview posts ALL subsequent SECTION_CLICK and FIELD_CHANGE to that stored parent
   origin (not to VITE_OS_URL).
4. OsPreview emits { type: "PREVIEW_READY" } back to the parent after init.
5. BrandWorkspace optionally records PREVIEW_READY for diagnostics.

VERIFY: BrandWorkspace's existing handleMessage origin guard
(e.origin === window.location.origin || e.origin === previewOriginRef.current)
must accept PREVIEW_READY — confirm previewOriginRef is set before the iframe can reply.
